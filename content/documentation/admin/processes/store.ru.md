Расширенное описание [хранилища процесса](./overview/#хранилище-процесса): правила записи, операции и просмотр при запуске.

## Откуда берутся данные

Данные попадают в хранилище двумя способами:

1. **После успешного внешнего действия (HTTP и др.)** — если в каталоге действия включено «Обновление хранилища» и заданы «Правила обновления хранилища». Значение для записи задаётся Go-шаблоном (можно использовать ответ действия, уже записанные ключи хранилища, параметры сущности).
1. **Задача с действием «Обновление хранилища процесса»** — отдельный тип действия без вызова внешней системы. Правила выполняются сразу при проходе задачи. Удобно для накопления JSON в циклах и условной записи.

### Переопределение на элементе задачи

На **элементе задачи** в редакторе процесса (блок «Правила хранилища») можно задать правила, отличные от каталога действия. При запуске используется такой порядок:

1. Правила на **элементе задачи** (если заданы).
1. Правила из **каталога действия**.
1. Правила, указанные в **конфигурации действия** (для «Обновление хранилища процесса»).

## Структура данных

Помимо простых строк хранилище поддерживает **JSON-объекты и массивы**. Ключ записи задаётся **путём через точку**, например:

| Путь | Значение |
| ---- | -------- |
| `notification.module_name` | Поле внутри объекта `notification` |
| `current_branch.scan_types.cve.findings` | Вложенное поле |
| `notification.branches` | Массив; для добавления элемента используйте операцию «Добавить JSON» |

Чтение в шаблонах: `{{ .store.notification.module_name }}`, `{{ .store._loop.item.branch }}` (текущий элемент цикла — см. [«Циклы и шаблоны»](./loops-and-templates/)).

Подробнее о шаблонах — в разделе [«Шаблонизация»](../../user/templating/#хранилище-процесса).

## Операции правил

В редакторе правил для каждой строки выбирается **операция**:

| Операция в интерфейсе | Назначение |
| --------------------- | ---------- |
| Записать строку | Текстовое значение по указанному пути |
| Записать JSON | Результат шаблона интерпретируется как JSON и записывается по пути |
| Добавить строку | Добавить строку в массив по пути |
| Добавить JSON | Добавить элемент в JSON-массив (например, очередная ветка в `notification.branches`) |
| Слить JSON (поверхностно) | Дополнить существующий JSON-объект полями из результата шаблона |
| Удалить | Удалить ключ по пути (поле «Источник» не используется) |

## Поле «Условие»

У каждого правила можно задать **«Условие»** — Go-шаблон, который должен дать «да» (`true` или `1`). Если условие пустое, правило применяется всегда.

Пример: записать значение только когда ветка engagement совпадает с текущей:

```text
{{ eq .store._loop.item.branch .store.current_branch_tag }}
```

Правила выполняются **по порядку** сверху вниз.

## Панель «Пути хранилища»

При настройке правил (в каталоге действий и на элементе задачи с «Обновлением хранилища процесса») справа доступна панель **«Пути хранилища»**:

- **Из этого процесса** — пути, которые уже используются в других задачах обновления хранилища и в элементах «Шаблон» на этой схеме.
- **Контекст цикла** — готовые фрагменты для текущего элемента цикла (`{{ .store._loop.item }}`, `{{ .store._loop.index }}` и т. д.).

Путь можно **нажать или перетащить** в поля «Цель», «Источник» или «Условие». После первого запуска процесса панель показывает **пример значений** из хранилища последнего запуска.

## Просмотр при запуске

В диалоге **«Запуск процесса»** откройте вкладку **«Хранилище»**:

| Режим | Описание |
| ----- | -------- |
| Дерево | Слева — ключи и типы данных; справа — полный JSON выбранного фрагмента без обрезки |
| JSON | Всё хранилище целиком, с прокруткой |

Кнопка **«Копировать»** помещает отображаемый JSON в буфер обмена.

## Ограничения и типичные ошибки

- Максимальный размер хранилища — **1 МБ** на один запуск процесса.
- Если в шаблоне указан ключ, которого ещё нет в хранилище, шаг завершится **ошибкой**. Убедитесь, что правила записи выполняются раньше по схеме.
- Повторная запись по тому же пути **заменяет** значение (кроме операций «Добавить…» и «Слить JSON»).

## Пример: накопление данных по ветке

Типовая последовательность (отчёт по нескольким веткам и типам сканирования):

1. Задача **«Инициализация ветки»** — записать тег текущей ветки и пустой каркас объекта ветки в хранилище.
1. Внутренний цикл по engagements — задачи обновления хранилища; лишние итерации отсекаются **эксклюзивным шлюзом** на схеме.
1. Задача **«Добавить ветку в отчёт»** — операция «Добавить JSON» в массив веток, затем «Удалить» для временных ключей.

После циклов элемент **«Шаблон»** формирует текст отчёта, следующая **задача** отправляет его во внешнюю систему.
