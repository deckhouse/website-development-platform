Процессы — механизм автоматизации сложных бизнес-сценариев, который позволяет создавать визуальные схемы выполнения действий с поддержкой условной логики, параллельного выполнения и обработки ошибок.

С помощью процессов можно, например:

- последовательно создавать проект в GitLab и настраивать его переменные;
- параллельно подготавливать namespace в Kubernetes и секреты в Vault перед развёртыванием приложения;
- обходить коллекции из хранилища в цикле, накапливать данные и формировать итоговое сообщение через элемент «Шаблон».

Процесс, как и действие, привязывается к одному или нескольким ресурсам и может быть запущен для сущности этих ресурсов.

## Конфигурация процесса

### Основная информация

Для создания процесса перейдите в раздел «Самообслуживание» → «Процессы» и нажмите кнопку «Создать».

| Название         | Обязательность | Описание                                                              |
| ---------------- | -------------- | --------------------------------------------------------------------- |
| Название         | Да             | Произвольное название процесса                                        |
| Описание         | Нет            | Подробное описание назначения процесса                                |
| Ресурс           | Нет            | Один или несколько ресурсов, для которых может запускаться процесс    |
| Владелец         | Нет            | Пользователь, ответственный за процесс                                 |
| Команда владелец | Нет            | Команда, ответственная за процесс                                      |
| Теги             | Нет            | Теги для категоризации процесса                                       |
| Иконка           | Нет            | Иконка для отображения в интерфейсе                                   |

### Схема процесса

Настройка схемы выполняется в визуальном редакторе на вкладке «Конфигурация».

{{< alert level="info" >}}
Для настройки процесса необходимо сначала заполнить основные поля и создать его. После этого станут доступны вкладки «Конфигурация» и «Параметры».
{{< /alert >}}

#### Добавление элементов

1. Добавьте элемент, выбрав его тип на панели редактора.
1. Настройте параметры элемента в боковой панели (действие, условия и др.).
1. Свяжите элементы друг с другом.

#### Элементы процесса

Процесс состоит из следующих типов элементов:

- «Начало» — точка входа в процесс.
- «Задача» — выполнение конкретного действия.
- «Эксклюзивный шлюз» — условное ветвление.
- «Параллельный шлюз» — объединение нескольких веток процесса в одну.
- «Цикл» — повторение части процесса (фиксированное число итераций или обход коллекции из хранилища).
- «Шаблон» — рендер Go-шаблона в хранилище процесса (без отправки наружу).
- «Таймер» — пауза или отложенное продолжение по расписанию.
- «Примечание» — текстовый блок на схеме.
- «Ошибка» — немедленная остановка процесса со статусом «Неудачно».
- «Конец» — завершение процесса.

##### Задача

Каждая задача — это запуск конкретного действия, созданного ранее в разделе «Самообслуживание» → «Действия».

##### Эксклюзивный шлюз

Эксклюзивный шлюз позволяет настраивать ветвление процесса по условиям. По умолчанию можно проверять статус предыдущей задачи и идти по «успешной» или «неуспешной» ветке.

Доступны два вида условий (можно комбинировать несколько правил через **AND** или **OR**):

| Вид                  | Пример                                           | Когда использовать                                              |
| -------------------- | ------------------------------------------------ | --------------------------------------------------------------- |
| Статус задачи        | `status == 'Completed'`                          | После задачи с внешним действием; подставляется статус предыдущей задачи на схеме |
| Значение из хранилища | `{{ .store._loop.item.scan_type }}` equals `cve` | Внутри цикла, фильтр по полю текущего элемента                  |
| Выражение «иначе»    | `true`                                           | Вторая исходящая ветка, если предикаты не сработали             |

{{< alert level="info" >}}
Предикаты по полям хранилища проверяются раньше веток с выражением `true`. Подробнее — в разделе [«Циклы и шаблоны»](./loops-and-templates/#эксклюзивный-шлюз-в-цикле).
{{< /alert >}}

Если условие выполняется, процесс идёт по соответствующей исходящей ветке (в редакторе — «успешная» / «неуспешная» линия).

##### Параллельный шлюз

Применяется для объединения нескольких веток в одну по заданным условиям.

В конфигурации параллельного шлюза можно настроить параметры ожидания:

- ожидать выполнения всех входящих элементов для продолжения выполнения;
- ожидать выполнения как минимум одного входящего элемента для продолжения выполнения.

При этом можно настроить, что именно считать «выполнением»:

- только успешное завершение входящих в шлюз задач;
- переход входящих задач в любой финальный статус («Неудачно», «Пропущено» и т. д.).

Параллельный шлюз также может использоваться для разделения веток, если подключается после элементов «Эксклюзивный шлюз» или «Цикл», либо в качестве вспомогательного элемента для повышения читаемости схемы процесса.

##### Цикл

Элемент «Цикл» предназначен для многократного выполнения одной части процесса; после завершения всех итераций активируется элемент на «Выход из цикла».

Режимы:

- «Фиксированное число» — поле «Количество итераций» (1–10000).
- «По коллекции» — поле «Шаблон коллекции»: Go-шаблон, возвращающий JSON-массив из хранилища или ответа (например `{{ toJSON .store.branch_tags }}`).

Подробнее — в разделе [«Циклы и шаблоны»](./loops-and-templates/).

Из элемента «Цикл» должны выходить ровно две связи к разным элементам:

1. «Тело цикла» — нижний порт; ветка, выполняемая на каждой итерации.
1. «Выход из цикла» — правый порт; продолжение после последней итерации.

Элемент «Цикл» можно ставить после «Эксклюзивного шлюза» или «Параллельного шлюза», если нужно повторять выбранную ветку, либо обходить массив из хранилища (engagements, теги веток и т. д.).

Типовой пример: цикл по коллекции → задачи «Обновление хранилища процесса» → выход → элемент «Шаблон» → задача отправки.

##### Шаблон

Элемент «Шаблон» записывает результат Go-шаблона в ключ хранилища (например `rendered_message`). Не выполняет запросов во внешние системы и не отправляет сообщения — только формирует текст. Отправка — следующей задачей.

Поля: «Тело шаблона», «Ключ в хранилище», опционально «Подсказка формата» (для UI).

Подробнее — в разделе [«Циклы и шаблоны»](./loops-and-templates/#элемент-шаблон).

##### Таймер

Элемент «Таймер» переводит экземпляр в ожидание до наступления заданной задержки или времени по расписанию. Пока таймер не сработал, статус может быть «Ожидание».

Подробнее — в разделе [«Циклы и шаблоны»](./loops-and-templates/#элемент-таймер).

##### Примечание

Элемент «Примечание» предназначен для текстовых пояснений на схеме процесса: он не выполняется при запуске и не соединяется с другими элементами.

В конфигурации примечания можно задать:

- «Текст» — содержимое с поддержкой Markdown-разметки;
- «Цвет фона» — цвет заливки фона;
- «Цвет текста» — цвет текста.

Примечание всегда позиционируется за остальными элементами и может использоваться для визуального выделения частей процесса. Также это единственный элемент, размер которого можно изменить.

##### Ошибка

Элемент «Ошибка» принудительно переводит запущенный процесс в статус «Неудачно»: при достижении этого элемента выполнение всех действий процесса прерывается.

От элемента не должно исходить связей к следующим элементам.

### Параметры процесса

На вкладке «Параметры» настраиваются параметры процесса, которые могут использоваться во всех действиях процесса.

Конфигурация и использование параметров процесса описаны в разделе [«Шаблонизация»](../../user/templating/#параметры-процесса).

### Хранилище процесса

Хранилище процесса — хранилище «ключ — значение» для передачи данных между шагами процесса. Поддерживаются строки и JSON (объекты, массивы); вложенные поля задаются путём через точку, например `notification.branches`, `current_branch.scan_types.cve.findings`.

- Одно хранилище на каждый запуск процесса.
- Запись — по правилам в действиях («Обновление хранилища» у внешних действий или тип действия «Обновление хранилища процесса») и по переопределению на элементе задачи.
- Чтение — Go-шаблоны `{{ .store.<ключ> }}` в действиях, шлюзах и элементе «Шаблон».

Подробности описаны в разделе [«Хранилище процесса»](./store/).

## Запуски процессов

### Запуск процесса

1. Перейдите к сущности, для которой нужно запустить процесс.
1. В меню сущности выберите «Запустить процесс».
1. Выберите нужный процесс из списка.
1. Заполните параметры процесса.
1. Нажмите «Запустить».

При запуске процесса доступны:

- «Общие параметры процесса» — параметры, определённые в конфигурации процесса;
- «Параметры действий» — параметры для каждого действия в процессе;
- «Переменные окружения» — дополнительные переменные для выполнения.

### Статусы процесса

Процесс может находиться в следующих статусах:

- «Создан» — процесс создан, но не запущен.
- «Выполняется» — процесс находится в процессе выполнения.
- «Ожидание» — процесс ждёт срабатывания таймера.
- «Приостановлен» — выполнение процесса приостановлено.
- «Завершен» — процесс успешно завершен.
- «Неудачно» — процесс завершился с ошибкой.
- «Отменен» — выполнение процесса было отменено.

### Управление активным процессом

Для активных процессов доступны следующие операции:

- «Приостановить» — временно остановить выполнение.
- «Возобновить» — продолжить выполнение после приостановки.
- «Остановить» — полностью остановить выполнение.
- «Принудительный перезапуск» — перезапустить процесс с начала.

### Отслеживание состояния

В разделе «Запуски процессов» можно просмотреть:

- список всех запусков процессов для сущности;
- детальную информацию о каждом запуске (диалог «Запуск процесса»);
- логи выполнения действий;
- статус каждого элемента процесса на схеме.

В диалоге запуска доступны вкладки:

| Вкладка     | Содержимое                                                                                    |
| ----------- | --------------------------------------------------------------------------------------------- |
| Схема       | Граф процесса, статусы узлов, лог выполнения                                                |
| Таймлайн    | Хронология событий экземпляра                                                                 |
| Хранилище   | Дерево ключей и полный JSON (см. [«Хранилище процесса»](./store/#просмотр-при-запуске)) |

#### Лог процесса

Для каждого запуска процесса доступен подробный лог его выполнения. Лог отображается внизу вкладки «Схема» (панель «Лог выполнения процесса»).

## Примеры использования

### Создание проекта с настройкой

1. «Начало» — запуск процесса.
1. «Задача» — создание проекта в GitLab.
1. «Эксклюзивный шлюз» — проверка успешности создания.
1. «Задача» (при успехе) — настройка переменных проекта.
1. «Задача» (при ошибке) — отправка уведомления об ошибке.
1. «Конец» — завершение процесса.

### Развертывание приложения

1. «Начало» — запуск процесса развертывания.
1. «Параллельный шлюз» — разделение на ветки.
1. «Задача» (ветка 1) — создание namespace в Kubernetes.
1. «Задача» (ветка 2) — создание секретов в Vault.
1. «Параллельный шлюз» — ожидание завершения обеих веток.
1. «Задача» — развертывание приложения.
1. «Конец» — завершение процесса.

## Ограничения

- Процессы не могут содержать более 100 элементов.
- Максимальное время выполнения процесса — 24 часа.
- Количество одновременных запусков процесса ограничено настройками системы.
- Некоторые действия могут быть недоступны для использования в процессах.
- Размер хранилища процесса — до 1 МБ на экземпляр; тело элемента «Шаблон» — до 1 МБ.
- Цикл — не более 10000 итераций (в обоих режимах).
